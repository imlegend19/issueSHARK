3.2. Application¶

class issueshark.issueshark.IssueSHARK[source]¶

Main application

  1. Connects to MongoDB

  2. Searches for the project in it

  3. Creates an issue system document, if not already in the database. If it is in database, it updates the last_updated field

  4. Finds fitting backend via find_fitting_backend()

  5. Calls process() on the found backend

_process_issue(jira_issue_id)[source]¶

Processes the issue in three steps:

  1. Get the issue with the given id from the jira rest api. See: _get_newest_issue()

  2. update the issue in the database (or store it if it was not parsed before). See: _store_jira_issue()

  3. Stores all events of this issue. See: _store_events()

  4. Stores all comments of this issue. See: _store_comments()

Parameters

jira_issue_id – id of the issue that is to be retrieved (e.g. “ZOOKEEPER-1”)

_process_issue(issue)[source]¶

Processes the issue in several steps:

  1. Get all comments. See: issueshark.backends.helpers.bugzillaagent.BugzillaAgent.get_comments()

  2. Get the whole issue history. See: issueshark.backends.helpers.bugzillaagent.BugzillaAgent.get_issue_history()

  3. Transforms the issue to our issue model. See: issueshark.backends.bugzilla.BugzillaBackend._transform_issue()

  4. Go through the history of the issue (newest to oldes) and store the events. See: issueshark.backends.bugzilla.BugzillaBackend._process_event()

  5. Process all comments. See: issueshark.backends.bugzilla.BugzillaBackend._process_comments()

Parameters

issue – issue that was got from the bugzilla REST API

process()[source]¶

Gets all the issues and their updates

  1. Gets the last stored issues updated_at field

  2. Gets all issues that was last change since this value

  3. Processes the results in 50-steps

  4. For each issue calls: issueshark.backends.bugzilla.BugzillaBackend._process_issue()
